Instalação Padrão do playwright:
npm init playwright@latest


Nosso projeto utilizará as seguintes abordagens:
- Separação dos cenários de negócio dos scripts de automação
    - Vamos precisar de no mínimo 2 arquivos
----> pasta features, que contém arquivos 
      .feature - funcionalidades e Cenários de negócio / Critérios de aceitação / Dados
----> pasta steps, que contém arquivos 
      .spec.js - Script de automação de teste
    Instalações adicionais: 
    //instala o pacote que dá suporte ao BDD
    //BDD = Desenvolvimento Direcionado por Comportamento
    npm install -D playwright-bdd


Feature: Login
    Scenario: Login with Sucess
        Given
        When

funcionalidade: Login

        Entao

Nomenclatura
- BDD - Behavior Driven Development - Desenvolvimento direcionado pelo Comportamento
  Abordagem de desenvolvimento, muito valorizada dentro da agilidade
  Objetivo é facilitar a comunicação entre os membros do time, áreas e clientes

- Gherkin: as instruções do BDD são escritas na sintaxe (formato) chamado Gherkin

- Cucumber/Behave/SpecFlow  são exemplos de plugins que fazem a integração entre arquivos .feature 
  com arquivos .spec.js

- Page Objects / Page Factory _ a ideia é separar o mapeamento dos elementos da interface gráfica 
(páginas, telas) do código da automação. 
---> Pages com mapeamento de elementos
---> Steps com os scripts de automação
---> Features com os cenários de teste

Em projeto simples, cada teste é um arquivo .spec.js diferente
Em um projeto com Gherkin, cada teste é composto por um arquivo . feature + um arquivo steps.js

Diferenças de sintaxe conforme o type definido no package.json
- commonjs --> quer que você use const ... = require('...')
- module --> quer que você faça import ... from ...

Sobre execução sincrona _ uma parte espera a outra / Ex. do telefone (existe uma promessa).
Sobre execução assincrona _ a parte que começa a conversa (emissora) não espera, continua fazendo outras coisas / Ex. da mensagem no whatsapp.

Sem async await _ é uma transação síncrona
Com async await _ é uma transação assincrona

Estamos usando dois frameworks:
1. Playwright _ controla o browser --> timeout - mensagem específica que pode explicar a causa
2. CucumberJS _ organização        --> timeout - mensagem genérica que não vai explicar a causa
